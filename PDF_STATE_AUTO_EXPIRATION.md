# Complete PDF State Lifecycle - Intelligent Auto-Expiration

## üéØ **REQUIREMENT**

**User's Insight:** "It's not necessary that user always explicitly rejects PDF by sending a negative message, the user could also just move on to the next query"

**Solution:** Automatically invalidate pending PDF offers when user moves on to ANY other message type.

---

## üìä **PDF STATE LIFECYCLE**

### **4 States:**

1. **`pending_pdf_request`** - PDF offered, awaiting user response
2. **`pdf_fulfilled`** - User said yes, PDF sent ‚úÖ
3. **`pdf_declined`** - User explicitly said no ‚ùå
4. **`pdf_expired`** - User moved on (new query/greeting/irrelevant) ‚è±Ô∏è

## üéØ **KEY BENEFITS**

### **1. Natural Conversation Flow**
- ‚úÖ User doesn't need to explicitly reject
- ‚úÖ Moving on = implicit rejection
- ‚úÖ No lingering old offers

### **2. No False Positives**
- ‚úÖ "when **can** a tenant..." ‚Üí NOT treated as rejection
- ‚úÖ Only checks rejection when state is `pending_pdf_request`
- ‚úÖ New queries invalidate old offers automatically

### **3. Clean State Management**
- ‚úÖ Every query gets its own PDF offer
- ‚úÖ Old offers don't interfere with new queries
- ‚úÖ Clear audit trail in logs

### **4. User-Friendly**
- ‚úÖ No pressure to respond to PDF offer
- ‚úÖ Can ask new questions freely
- ‚úÖ PDF offer doesn't block conversation
